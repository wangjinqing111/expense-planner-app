# Bug 修复工作流

## 工作流程步骤

### 1. 问题排查

首先彻底理解问题：

1. **收集信息**：向用户获取错误信息、崩溃日志，或描述预期与实际行为的差异
2. **探索相关代码**：使用 Read、Grep、Glob 查找问题代码
3. **定位根本原因**：追溯问题的源头，而非表面现象
4. **形成假设**：记录你认为导致 bug 的原因

如果用户未提供错误详情，请在继续之前询问。

### 2. 修改计划

确定根本原因后：

1. **清晰描述问题**：在修复历史文件中清楚说明问题
2. **提出解决方案**：解释修复方法
3. **确定受影响文件**：列出所有需要修改的文件
4. **规划变更内容**：概述具体修改方案

### 3. 测试验证

实现修复后：

1. **运行相关测试**：使用 `flutter test` 针对相关区域
2. **验证修复**：如果可能，运行应用或相关代码确认问题已解决
3. **检查回归**：确保修复不会破坏其他功能
4. **静态分析**：使用 `flutter analyze` 检查问题

### 4. 提交代码

**重要：提交前，务必将修复历史保存到 `./z_guides/fix/`**

1. **创建修复历史文件**：
   - 位置：`./z_guides/fix/`
   - 文件名格式：`YYYY-MMDD-问题简要描述.md`
   - 示例：`2026-4-22-add_expense_screen_null_check.md`

2. **修复历史文件内容**：
```markdown
# Bug 修复记录

## 日期
YYYY-MM-DD HH:MM

## 问题描述
[清晰描述 bug]

## 根本原因
[什么导致了 bug]

## 解决方案
[如何修复]

## 修改的文件
- file1.dart
- file2.dart

## 验证方式
[如何验证修复 - 测试通过、手动测试结果等]
```

3. **提交变更**：
   - 仅暂存相关文件（避免暂存无关变更）
   - 按照项目规范创建提交信息

## 修复历史存储

所有修复历史文件存放在 `./z_guides/fix/` 目录。每个修复对应一个文件，文件名包含时间戳和简要问题描述。这为项目创建了可搜索的 bug 修复历史。

## 注意事项

- 如果用户提供了崩溃日志或错误信息，务必将其包含在修复历史中
- 对于复杂的 bug，可以先创建计划再进行修改
- 提交前务必验证修复有效
- 保持修复历史文件专注 - 一个 bug 对应一个文件
